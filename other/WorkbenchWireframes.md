# AI 工作台 Demo 低保真 Wireframe 说明

基于 `other/WorkbenchDemo.md`，这份文档继续下钻到页面级低保真结构，目的是让产品、设计、前端对页面骨架、信息层级和关键交互形成一致理解。

说明：
- 这是低保真说明，不是视觉稿。
- 重点是区块结构、信息优先级、交互流。
- 默认角色：`区域运营负责人`
- 默认布局：`左侧导航 240 / 中间主内容自适应 / 右侧功能栏 320`

---

## 1. 全局框架

### 1.1 通用页面骨架
所有页面共用同一套三栏框架：

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Top Bar: Logo / 产品名 / 角色 / 管理范围 / 日期 / 全局搜索 / 用户信息       │
├──────────────┬───────────────────────────────────────────────┬──────────────┤
│ Left Nav     │ Main Workspace                                │ Right Panel  │
│              │                                               │              │
│ Mode Switch  │ Page Header                                   │ Tab:         │
│ Main Nav     │ Main Content                                  │ [任务][提醒][报告]│
│ Recent Chats │                                               │              │
│ Quick Prompts│                                               │ Tab Content  │
└──────────────┴───────────────────────────────────────────────┴──────────────┘
```

### 1.2 全局固定元素

#### Top Bar
建议所有页面固定：
- 左：产品名称 `AI 工作台`
- 中：角色和范围
  - `区域运营负责人`
  - `华东区 · 182 家门店`
- 右：
  - 日期选择
  - 全局搜索
  - 消息/通知
  - 用户头像

#### Left Nav
固定区块：
1. 模式切换
   - `AI 工作台`
   - `传统模式`
2. 主导航
   - 今日工作台
   - 经营分析
   - 门店风险
   - 任务协同
   - 报告中心
3. 最近会话
4. 快捷提问

#### Right Panel
固定 Tab：
- 任务
- 提醒
- 报告

默认行为：
- 首页默认 `提醒`
- 经营分析页默认 `报告`
- 门店风险页默认 `任务`
- 任务协同页默认 `任务`
- 报告中心页默认 `报告`

## 2. 页面 1：AI 工作台首页

### 2.1 页面目标
首页不是空白输入框，而是“今天的工作入口”。
用户一进来要立刻知道：
- 今天发生了什么
- 我应该先看什么
- 我可以直接问什么

### 2.2 Wireframe 结构

```text
┌──────────────────────── Main Workspace ────────────────────────┐
│ Page Header                                                    │
│ 标题：今日工作台                                                │
│ 副信息：华东区 / 182 家门店 / 2026-03-20                        │
├────────────────────────────────────────────────────────────────┤
│ Summary Strip                                                  │
│ [高风险门店 3] [待处理任务 5] [新报告 1] [命中提醒 4]            │
├────────────────────────────────────────────────────────────────┤
│ Hero Card: 今日 AI 摘要                                         │
│ - 利润率下降 1.6pct                                             │
│ - 5 家新店差评异常                                               │
│ - 2 家门店库存告警                                               │
│ [为什么利润下降] [查看异常门店] [生成整改任务]                  │
├────────────────────────────────────────────────────────────────┤
│ Recommended Questions                                          │
│ [华东区昨天利润率为什么下降？]                                 │
│ [新店差评超过 3 条的有哪些？]                                  │
│ [帮我建一个外卖订单下滑预警]                                   │
├────────────────────────────────────────────────────────────────┤
│ Default Conversation Thread                                    │
│ User: 昨天华东区利润率为什么下降？                              │
│ AI: 一句话结论 + Top 3 归因 + 继续追问                          │
└────────────────────────────────────────────────────────────────┘
```

### 2.3 区块说明

#### A. Page Header
- 左：`今日工作台`
- 右：时间范围切换 `昨天 / 近7天 / 自定义`
- 不要塞太多筛选条件，首页先讲故事

#### B. Summary Strip
用 4 个 mini KPI 卡片概括今天最重要的工作量：
- 高风险门店数
- 待处理任务数
- 新报告数
- 命中提醒数

每个卡片可点击跳转：
- 高风险门店 -> 门店风险页
- 待处理任务 -> 任务协同页
- 新报告 -> 报告中心页
- 命中提醒 -> 右侧提醒 Tab

#### C. Hero Card
首页视觉中心。
这里必须像“晨会摘要”，而不是运营看板。

卡片结构：
- 标题：`今天你最该关注的 3 件事`
- 结论列表：3 条以内
- 底部动作按钮：直接跳后续页面或发起对话

#### D. Recommended Questions
用按钮形式展示，不用自然语言输入提示文案堆满页面。
建议 3 个以内。

#### E. Default Conversation Thread
首页下半屏保留一个真实业务对话流，目的是：
- 让用户感知 AI 是怎么工作的
- 避免首页像单纯 dashboard

内容长度控制：
- 1 个用户问题
- 1 个 AI 回答卡
- 2 个追问建议

### 2.4 右侧 Tab 内容
首页默认打开 `提醒`：

```text
[提醒]
- 外卖订单下滑预警：4 家门店
- 新店差评异常：5 家门店
- 库存低于阈值：2 家门店
[查看全部]
```

### 2.5 关键交互
- 点 Hero Card 按钮 -> 打开分析页或风险页
- 点推荐问题 -> 在中间发起对话
- 点右侧提醒项 -> 进入对应上下文页面

## 3. 页面 2：经营分析页

### 3.1 页面目标
把“AI 查数”和“AI 分析”讲清楚。
这个页面必须看起来像：
- 已经完成了一次高质量分析
- 而不是聊天记录堆叠

### 3.2 Wireframe 结构

```text
┌──────────────────────── Main Workspace ────────────────────────┐
│ Page Header                                                    │
│ 标题：经营分析                                                  │
│ 子标题：围绕当前问题的归因与建议                                 │
├────────────────────────────────────────────────────────────────┤
│ Query Bar                                                      │
│ User Query: 华东区昨天利润率为什么下降？                         │
│ [修改问题] [保存为报告] [生成任务]                              │
├────────────────────────────────────────────────────────────────┤
│ Conclusion Card                                                │
│ 一句话结论                                                      │
│ 利润率下降主要由包装成本上升与活动补贴过深导致                   │
│ [确定性数据] [AI建议] 标签分区                                  │
├────────────────────────────────────────────────────────────────┤
│ Insight Cards                                                  │
│ [归因1] [归因2] [归因3]                                         │
├────────────────────────────────────────────────────────────────┤
│ Visualization Area                                             │
│ 左：趋势图                                                     │
│ 右：区域/门店分布图                                             │
├────────────────────────────────────────────────────────────────┤
│ Detail Table                                                   │
│ 门店 / 利润率 / 包装成本率 / 活动补贴 / 差评率 / 风险等级         │
├────────────────────────────────────────────────────────────────┤
│ Follow-up Questions                                            │
│ [展开看南京门店] [只看外卖渠道] [生成运营整改任务]               │
└────────────────────────────────────────────────────────────────┘
```

### 3.3 区块说明

#### A. Query Bar
这一行要清晰表达：
- 当前正在回答什么问题
- 用户还能做哪些下一步动作

按钮建议：
- `修改问题`
- `保存为报告`
- `生成任务`

#### B. Conclusion Card
这是整个页面视觉焦点。

建议结构：
- 标题：`结论`
- 一句话结论
- 影响范围
- 优先动作

这里必须把“确定性数据结论”和“AI 推断建议”分开。
否则会让用户对可信度没有判断。

#### C. Insight Cards
三张并列卡：
- 归因 1：包装成本率上升
- 归因 2：活动补贴过深
- 归因 3：两家门店配送补偿偏高

每张卡结构：
- 指标变化
- 影响对象
- 是否异常

#### D. Visualization Area
避免图太多。
建议只保留两个：
- 趋势图：近 7 日利润率与成本率走势
- 分布图：问题门店分布/区域对比

#### E. Detail Table
给用户“往下钻”的能力。
表格字段只留最必要字段，避免像 BI 工具。

#### F. Follow-up Questions
这块很重要。
不能让用户看完分析没路走。
必须给：
- 往下钻
- 转动作
- 转沉淀

### 3.4 右侧 Tab 内容
经营分析页默认打开 `报告`：

```text
[报告]
- 今日分析简报（草稿）
- 本周经营复盘
- 新店口碑报告
[保存当前分析为报告]
```

### 3.5 关键交互
- `保存为报告` -> 报告中心生成新草稿
- `生成任务` -> 打开任务预览弹层
- `展开看南京门店` -> 原地刷新下方表格与图表

## 4. 页面 3：门店风险页

### 4.1 页面目标
展示 AI 主动发现问题的能力。
这个页面要像“风险待处理台”，不是列表查询页。

### 4.2 Wireframe 结构

```text
┌──────────────────────── Main Workspace ────────────────────────┐
│ Page Header                                                    │
│ 标题：门店风险                                                  │
│ 子标题：系统自动识别的高优先级问题                               │
├────────────────────────────────────────────────────────────────┤
│ Filter Bar                                                     │
│ [区域] [风险类型] [风险等级] [时间范围] [只看我负责门店]          │
├────────────────────────────────────────────────────────────────┤
│ Risk Overview                                                  │
│ [高风险 12] [中风险 24] [今日新增 6] [已闭环 8]                  │
├────────────────────────────────────────────────────────────────┤
│ Risk List                                                      │
│ ┌────────────────────────────────────────────────────────────┐ │
│ │ 门店A | 高风险 | 差评集中在出餐慢 | 建议：检查排班        │ │
│ │ 门店B | 高风险 | 食安相关评价增多 | 建议：立即巡检        │ │
│ │ 门店C | 中高风险 | 新品退单异常 | 建议：复核制作标准      │ │
│ └────────────────────────────────────────────────────────────┘ │
├────────────────────────────────────────────────────────────────┤
│ Risk Detail Drawer / Panel                                     │
│ 选中门店后展开：                                                │
│ - 风险摘要                                                      │
│ - 可能原因                                                      │
│ - 相关指标                                                      │
│ - 建议动作                                                      │
│ - [生成任务] [加入巡检计划]                                     │
└────────────────────────────────────────────────────────────────┘
```

### 4.3 区块说明

#### A. Filter Bar
这是风险页必须要有的，因为运营会按责任范围筛选。
但字段不要太多，只保留：
- 区域
- 风险类型
- 风险等级
- 时间范围
- 是否仅看自己负责范围

#### B. Risk Overview
4 张概览卡足够：
- 高风险
- 中风险
- 今日新增
- 已闭环

#### C. Risk List
列表比大卡片更适合 Demo，因为信息密度高且容易扫读。

每行建议字段：
- 门店名
- 风险等级
- 问题摘要
- 推荐动作
- 来源标签

#### D. Risk Detail Drawer
点击某门店展开详情，而不是整页跳转。
这样更适合现场讲 demo。

抽屉结构：
- 风险标题
- 风险描述
- 最近 7 日相关指标
- 原始评价/事件摘要
- 建议动作
- CTA：生成任务 / 加入巡店计划

### 4.4 右侧 Tab 内容
门店风险页默认 `任务`：

```text
[任务]
- 待创建任务：3
- 今日已下发：5
- 即将超时：2
[批量生成任务]
```

### 4.5 关键交互
- 勾选多家门店 -> `批量生成任务`
- 点击某风险项 -> 展开明细抽屉
- 点击 `加入巡检计划` -> 生成对应任务或路线计划

## 5. 页面 4：任务协同页

### 5.1 页面目标
展示“从 AI 洞察到执行闭环”的关键价值。

### 5.2 Wireframe 结构

```text
┌──────────────────────── Main Workspace ────────────────────────┐
│ Page Header                                                    │
│ 标题：任务协同                                                  │
│ 子标题：AI 生成与人工创建的执行任务                              │
├────────────────────────────────────────────────────────────────┤
│ Task Summary                                                   │
│ [待处理 18] [处理中 7] [即将超时 3] [待确认 4]                  │
├────────────────────────────────────────────────────────────────┤
│ Task Filter Bar                                                │
│ [任务来源] [状态] [责任人] [区域] [截止时间]                     │
├────────────────────────────────────────────────────────────────┤
│ Task List + Detail Split View                                  │
│ ┌────────────────────┬──────────────────────────────────────┐ │
│ │ 任务列表            │ 任务详情                              │ │
│ │                    │                                      │ │
│ │ 任务A              │ 标题：整改新店差评问题                │ │
│ │ 来源：AI分析       │ 来源：门店风险页                      │ │
│ │ 状态：待处理       │ 责任人：南京区域经理                  │ │
│ │ 截止：明日18:00    │ 背景：近2日差评率持续上升             │ │
│ │                    │ 建议动作：检查排班/复核打包流程        │ │
│ │                    │ 提交要求：上传整改结果                 │ │
│ │                    │ 回看节点：7天后自动复盘               │ │
│ │                    │ [标记处理中][转派][失效][查看回看]     │ │
│ └────────────────────┴──────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

### 5.3 区块说明

#### A. Task Summary
4 个核心状态卡：
- 待处理
- 处理中
- 即将超时
- 待确认

#### B. Task Filter Bar
保留执行管理最常见的维度：
- 来源
- 状态
- 责任人
- 区域
- 截止时间

#### C. Split View
任务页最适合左右分栏：
- 左边列表，便于扫描多个任务
- 右边详情，便于讲清单个任务闭环

#### D. Task Detail
详情里一定要有这 5 块：
1. 问题背景
2. 关联指标
3. 建议动作
4. 提交要求
5. 回看节点

因为这是“闭环”成立的最小结构。

### 5.4 右侧 Tab 内容
任务页仍然默认 `任务`，但做更轻量摘要：

```text
[任务]
- 我负责的任务：6
- 今日新建：4
- 即将超时：3
- 最近完成：2
```

### 5.5 关键交互
- `标记处理中`
- `转派`
- `失效`
- `查看回看`
- `从当前任务生成周报素材`

## 6. 页面 5：报告中心页

### 6.1 页面目标
把“稳定输出”和“角色化报告”做成可见能力。

### 6.2 Wireframe 结构

```text
┌──────────────────────── Main Workspace ────────────────────────┐
│ Page Header                                                    │
│ 标题：报告中心                                                  │
│ 子标题：日报、周报与专题分析                                     │
├────────────────────────────────────────────────────────────────┤
│ Report Filter Bar                                              │
│ [报告类型] [角色模板] [时间范围] [状态]                         │
├────────────────────────────────────────────────────────────────┤
│ Report Card Grid / List                                        │
│ [全国经营简报] [华东经营复盘] [新店口碑周报] [活动效果分析]      │
├────────────────────────────────────────────────────────────────┤
│ Selected Report Detail                                         │
│ 标题 / 生成时间 / 推送对象 / 摘要 / 关键结论 / 指标明细 / 建议动作 │
├────────────────────────────────────────────────────────────────┤
│ Subscription Section                                           │
│ [已订阅] 老板每日8点全国简报                                    │
│ [已订阅] 华东区周一9点经营复盘                                  │
│ [新建订阅]                                                     │
└────────────────────────────────────────────────────────────────┘
```

### 6.3 区块说明

#### A. Report Filter Bar
建议只保留：
- 报告类型：日报 / 周报 / 专题
- 角色模板：老板 / 区域运营 / 督导
- 时间范围
- 状态：草稿 / 已发送 / 已订阅

#### B. Report Card Grid
可以用卡片流，也可以用列表。
Demo 里推荐卡片流，更容易讲：
- 标题
- 摘要
- 最近更新时间
- 状态标签

#### C. Selected Report Detail
点击报告卡后，在下半区展示详情，不需要新跳页。
这样便于现场连续演示。

详情结构：
- 报告标题
- 面向角色
- 推送对象
- 核心结论 3 条
- 指标模块
- 建议动作

#### D. Subscription Section
放在页面底部或右下角，展示报告不是一次性的，而是可订阅的。

### 6.4 右侧 Tab 内容
报告中心默认 `报告`：

```text
[报告]
- 最新生成：2
- 已订阅：4
- 今日待发送：1
[新建订阅]
```

### 6.5 关键交互
- `保存为草稿`
- `立即推送`
- `设为订阅模板`
- `复制一份给其他区域`

## 8. 右侧功能栏低保真说明

### 8.1 通用结构

```text
┌────── Right Panel ──────┐
│ [任务] [提醒] [报告]    │
├─────────────────────────┤
│ 当前 Tab 内容            │
│ - 列表项1                │
│ - 列表项2                │
│ - 列表项3                │
│                         │
│ [查看全部] / [新建]      │
└─────────────────────────┘
```

### 8.2 设计原则
- 右侧永远比中间轻
- 一屏最多显示 3~5 条
- 强调“沉淀结果”，不和中间抢叙事

### 8.3 各 Tab 展示最小字段

#### 任务
- 标题
- 状态
- 截止时间

#### 提醒
- 标题
- 命中规则/原因
- 影响对象

#### 报告
- 标题
- 更新时间
