## 1. 文档目的
基于 `docs/interview` 下 5 份访谈材料，提炼适用于 "agent as a service" 产品规划的客户共性需求，作为后续产品定义、方案拆解和 MVP 排期的统一输入。

## 3. 结论先行
从几家连锁餐饮客户的访谈看，客户并不需要一个泛化的 "会聊天的 AI"，而是需要一个 **基于可信数据、可嵌入现有管理流程、能形成经营与执行闭环的企业级 Agent 服务**。

客户的真实优先级高度一致：
1. 先解决 "查数慢、跨系统、靠人拉表" 的问题。
2. 再解决 "发现问题晚、靠人盯、不会主动提醒" 的问题。
3. 然后解决 "只看见问题，无法分发任务、跟踪执行、验证结果" 的问题。
4. 最后才是高风险的代操作自动化。

换句话说，客户购买的不是模型能力本身，而是：
- 安全可信的数据接入
- 稳定可控的工作流
- 面向管理角色的固定输出
- 与门店/督导/运营协同系统打通后的闭环价值

## 4. 客户共性需求

### 4.1 高优先级需求

#### 4.1.1 自然语言取数与统一数据入口
**共性现状**
- 数据散落在多个系统：美团管家、财务、供应链、会员、点评/外卖、内部档案系统等。
- 现有分析工作大量依赖人工导数、拼表、清洗、汇总。
- 管理层没有时间自己翻系统，更希望直接得到结果。

**核心诉求**
- 用自然语言直接提问并拿到准确结果。
- 支持跨系统、跨表、跨维度查询。
- 将常用查询沉淀成标准问法/标准模板，减少自由发挥带来的误差。
- 成为总部管理者统一的看数入口，减少多系统来回切换。

**典型问题**
- 新店近 30 天好评/差评及明细
- 各渠道营收占比、趋势、异常原因
- 会员复购率、会员销售占比、渠道贡献
- 菜品/活动/评价对营收或利润的影响
- 行业趋势、竞品动作、商圈变化

#### 4.1.2 主动监控、预警与订阅推送
**共性现状**
- 现在主要是人去看报表，问题发现滞后。
- 日常管理依赖经验和人工盯盘，效率低且容易漏看。

**核心诉求**
- AI 7x24 主动盯关键指标，而不是等人来查。
- 支持日/周/月报、异常提醒、阈值配置、定时推送。
- 推送到企业微信、钉钉、飞书、APP 等现有渠道。
- 同一类场景输出格式稳定，便于老板和管理层长期使用。

**重点预警对象**
- 营收、利润率、订单量、客单价
- 差评率、食安红线、服务风险
- 热销品库存/备货异常
- 活动效果异常、渠道表现异常
- 目标未达成、趋势偏离、重点门店异动

#### 4.1.3 经营分析要有结论，不是只堆数据
**共性现状**
- 客户不缺报表，缺的是 "问题是什么、原因是什么、应该怎么做"。
- 目前很多分析依赖经验判断，系统无法快速辅助决策。

**核心诉求**
- 报告必须给总结结论，不能把判断工作再留给管理者。
- 需要对指标异常做拆解、归因和优先级排序。
- 要支持从总部 -> 区域 -> 门店 -> 渠道 -> 菜品/评价/活动 的逐层下钻。
- 结论最好结合企业 SOP 或历史处理经验给出整改建议。

**期望输出形态**
- 固定模板的日报/周报/月报
- 问题描述 + 可能原因 + 建议动作
- Top N 关键问题，而不是过长的看板和菜单

#### 4.1.4 任务闭环与执行协同
**共性现状**
- 发现问题后，后续靠人拉群、发消息、盯执行，链路容易断。
- 督导、运营、店长之间缺少统一闭环视图。

**核心诉求**
- 经营洞察结果可直接生成待办或任务。
- 任务可分发给总部运营、区域经理、督导、店长等不同角色。
- 能跟踪状态、超时、失效、执行反馈。
- 执行完成后自动回看指标变化，验证动作是否有效。

**本质需求**
客户需要的不是单点 AI 分析，而是 "发现问题 -> 分派动作 -> 跟踪执行 -> 结果验证" 的闭环系统。

### 4.2 中优先级需求

#### 4.2.1 代操作自动化
**适用场景**
- 菜品上下架
- 改营业时间、改价格
- 营销活动创建与配置
- 批量商品/供应链信息上传
- 差评回复生成、申诉辅助
- 常见工单自动接单与标准化处理

**客户态度**
- 明确认可价值，但普遍认为风险高于查数。
- 更适合在低风险、标准化、规则明确的流程里先落地。
- 一旦涉及金额、价格、库存、营业状态等关键字段，必须可审核、可回滚、可记录。

#### 4.2.2 工单与服务台型 Agent
**适用客户**
- 门店数多、支持团队小、跨时区运营的品牌。

**核心诉求**
- 接入工单系统，优先拦截常见问题。
- 多轮问答无法解决时自动转人工。
- 保留完整工单记录，便于问题分类、统计和知识沉淀。
- 对需要现场动作的问题，支持视频/图文引导。

#### 4.2.3 行业对标与外部数据融合
**核心诉求**
- 能查看行业趋势、竞品动作、商圈变化。
- 支持连续追问和对标分析，而不只是静态报表。

**限制**
- 该能力对客户有吸引力，但不是最先购买的理由。
- 前提是内部经营数据已先跑通、且结果可信。

## 5. 客户共同的非功能要求

### 5.1 数据准确性要求极高
这是所有访谈里最一致的要求。

- 取数层面接近或达到 100% 准确，尤其财务、退款、经营口径类数据。
- 分析层允许是辅助参考，但必须明确哪些是计算结果、哪些是模型建议。
- 不接受模型直接编 SQL 后输出不可信结果。
- 更适合采用 "标准意图识别 + 固化查询模板 + 受控计算引擎"，而不是完全自由生成。

### 5.2 数据安全与权限控制
- 客户普遍不接受数据落到个人服务器或不透明的第三方环境。
- 对爬虫式抓取、账号复用、扫码登录、验证码中断等方案有强烈顾虑。
- 必须有企业级托管、权限隔离、访问审计和账号机制。
- 总部、区域、督导、门店等角色看到的数据范围必须不同。

### 5.3 稳定性与可重复性
- 同一任务、同一报表、同一问法，结果和格式必须稳定。
- 定时任务、消息发送、附件拉取、自动化执行不能频繁失败。
- 客户对 "偶尔能用" 的 AI 容忍度极低，必须能进入正式业务流。

### 5.4 嵌入现有系统，而非新建孤岛
- 客户不希望再多一个独立系统。
- 更倾向于嵌入现有美团管家入口，或与现有 POS、督导、工单等系统打通，而不是再切换到一套全新的后台。
- 更希望沿用现有总部、区域、督导、门店等管理链路，以及已有的任务承接和消息触达方式。

### 5.5 输出要分角色、分层级
不同角色关注点明显不同：
- 老板/高管：整体经营概览、关键异常、结论和优先级
- 运营/区域负责人：渠道、活动、菜品、目标达成、区域对比
- 督导：问题门店、可能原因、整改建议、任务跟踪
- 门店/店长：具体待办、执行动作、操作提示

## 6. 典型用户画像与关键场景

### 6.1 总部管理者
**目标**
- 快速掌握品牌经营健康度
- 看最重要的 3-5 个问题
- 做经营决策

**需要的 Agent 能力**
- 总览驾驶舱
- 异常归因
- 行业/竞品参考
- 周月报自动推送

### 6.2 运营/区域负责人
**目标**
- 找到影响业绩的具体原因
- 快速定位区域、渠道、门店问题
- 调整活动、菜品、投放与运营策略

**需要的 Agent 能力**
- 多维分析下钻
- 预警订阅
- SOP 建议
- 任务协同

### 6.3 督导/巡检角色
**目标**
- 知道该盯哪些门店、哪些问题优先处理
- 获得标准化整改建议
- 跟进任务执行结果

**需要的 Agent 能力**
- 固定格式日报
- 风险门店排序
- 任务清单
- 历史相似案例/SOP 匹配

### 6.4 门店/店长/一线人员
**目标**
- 接收明确待办
- 完成标准化操作
- 在必要时获得辅助而不是复杂分析

**需要的 Agent 能力**
- 简单明确的指令
- 低风险代操作
- 工单问答/视频指导

## 7. 对产品定义的启示

### 7.1 产品不应被定义为通用聊天机器人
更准确的定义应是：
> 一个面向连锁餐饮总部与门店协同的企业级 Agent 平台，基于可信数据、受控工作流和角色化输出，完成经营洞察与执行闭环。

### 7.2 推荐能力架构
建议至少拆成四层：
1. **数据接入层**：打通美团管家、POS、供应链、财务、会员、评价、工单等数据源。
2. **受控能力层**：标准查询模板、指标口径、规则引擎、权限系统、审计日志。
3. **Agent 编排层**：查数 Agent、预警 Agent、分析 Agent、工单 Agent、代操作 Agent。
4. **业务闭环层**：报告、消息推送、任务下发、执行反馈、效果验证。

### 7.3 推荐先后顺序
**Phase 1：可信查数 MVP**
- 自然语言取数
- 标准问法模板
- 固定格式日报/周报
- 会员或评价场景试点

**Phase 2：主动预警与分析**
- 异常发现
- 原因拆解
- SOP 建议
- 行业对标

**Phase 3：任务闭环**
- 自动生成任务
- 跟踪执行
- 验证效果

**Phase 4：代操作自动化**
- 从低风险高频操作开始
- 全链路加审核、回滚、审计

## 8. MVP 建议
结合访谈，最可行的 MVP 不是一次性做全，而是先打一个最小可闭环版本：

### 8.1 MVP 目标
面向总部用户，先证明三件事：
- 数据能准
- 报告能看
- 异常能推

### 8.2 MVP 建议范围
- 单一数据域切入：会员、评价或营业报表三选一
- 单一入口：嵌入美团管家 PC 端
- 单一输出：固定格式日报/周报 + 对话查数
- 单一动作：异常提醒，不直接自动执行高风险操作

### 8.3 MVP 成功标准
- 目标查询命中率高
- 关键报表准确率可验证
- 输出格式稳定
- 管理层每周有真实使用频次
- 相比人工拉数，时效显著提升

## 9. 明确不建议当前阶段优先投入的方向
- 完全开放式、自由生成 SQL 的分析助手
- 无审计、无权限控制的代操作 Agent
- 脱离现有系统的新独立后台
- 过于追求全场景覆盖而迟迟不能上线的 "大而全" 方案

## 10. 一句话产品要求
客户要的不是一个聪明的 AI，而是一个 **可靠、可控、可嵌入业务、能把数据洞察变成执行结果的 Agent 服务系统**。
