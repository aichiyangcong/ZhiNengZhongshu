一、会议背景

柠季团队目前已在内部尝试使用 Open Cloud（Claude）类 AI 工具处理日常工作，但在实际落地中遇到数据安全、稳定性、准确性等多方面问题，迫切希望与美团SaaS团队探讨：能否基于美团管家的数据能力，提供一套安全、可控、面向企业级用户的 AI 工具解决方案。

2.2 遇到的核心问题

问题一：操作不稳定

AI执行定时任务时，频繁用错账号（用个人账号而非机器人账号）发送消息

无头浏览器下载文件失败，坚持不使用指定的正常浏览器

登录遇验证码时任务中断，需人工干预

问题二：数据准确性达不到要求

让AI写SQL取数、自行计算分析，准确率极低（远低于90%）

试用过多个大模型（含国内外主流模型），均存在写错SQL、计算出错的问题

火山方面表示在预设条件极清晰时可达95%，但自然语言场景下仍无法保证

涉及财务数据的业务（如退款金额）必须100%准确，AI无法满足

问题三：数据安全与合规风险

数据存放在个人云服务器上，存在泄露隐患

通过页面爬取美团数据属于类爬虫行为，担心触发美团风控机制

不确定美团是否允许此类自动化操作

问题四：数据获取链路过长

想新增一个字段的分析，需要：找接口 → 联系美团开发开放字段 → 自方开发接入 → 搭建BI看板，全流程耗时极长

老板需求变化频繁，开发速度完全跟不上

三、柠季核心需求梳理

3.1 最高优先级：自然语言取数

痛点描述： 各部门（会员、营运、督导、校区等）日常大量时间消耗在"扒数据"上——到不同系统后台筛选、下载、合并多张表格，再人工处理，才能得到想要的分析维度。营支岗位（每个地区配备）的主要工作就是跨系统拉取数据并汇总。

希望实现的能力：

通过自然语言直接提问，AI能从美团管家后台取出对应数据（支持跨表、多维度）

支持设置阈值和定时任务，自动推送异常数据

数据准确率必须100%（取数层面，不是分析层面）

典型查询场景举例：

建店不超过30天的新门店，昨日好评数、差评数及明细

会员消费在各渠道（堂食/外卖/团购/抖音）的占比分布

某校区门店的订单到手率、折扣率趋势

线上线下营收对比及环比变化

关于准确率的共识（重要）：

取数准确率要求100%

若通过自然语言匹配"标准问"的方式（即把用户泛化问题匹配到固化查询模板），可接近100%

分析/计算层面由于模型幻觉不可避免，定位为"辅助参考"而非"权威结论"

不适合用AI做的场景：涉及精确计算（复购率、退款金额等），除非查询条件完全固化

3.2 次优先级：代操作（重复性任务自动化）

典型场景：

菜品上下架（每月1~2次，流程标准但繁琐）

批量上传供应链商品信息

营销活动创建与配置

现有顾虑：

操作类风险更高，担心AI出错（金额、参数填写错误）

需先在"查数"场景跑通后再推进

当前美团侧进展：美团如意（面向门店的AI助手）已覆盖菜品管理、会员、对账、营业报表查询等功能，通过意图识别+固化工作流实现，准确率较高。预计4月中旬后可面向门店开放。

五、关键讨论：经营诊断闭环产品设想

美团SaaS团队分享了一个正在规划的产品方向，获得柠季认可：

核心逻辑：发现问题 → 定位问题 → 分析问题 → 生成建议 → 任务分发 → 执行反馈 → 数据验证

具体实现路径：

智能诊断：基于每日经营指标（营收、人效、好评率、渠道占比等），自动识别异常

横向拉通：结合行业大盘数据，判断是普遍问题还是门店个性问题

深度下钻：按区域→门店→渠道→具体指标逐层定位根因

SOP匹配：结合企业内部培训知识库，生成针对性整改建议

任务闭环：通过督导系统下发整改任务，设定完成时间，跟踪完成状态

数据验证：一周后自动回溯相关指标，验证整改效果

柠季评价：方向正确，但建议先出一个MVP版本让企业跑起来，不要等完整版，模型能力持续迭代即可。
