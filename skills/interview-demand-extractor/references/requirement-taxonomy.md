# Requirement Taxonomy

Use this taxonomy when extracting needs from `docs/interview`.

## 1) 业务目标与成功指标
- 业务北极星（营收/利润/效率/服务质量）
- 管理目标（时效、准确性、可复制、扩张支撑）
- 成功判定指标（准确率、响应时长、闭环率、采纳率）

## 2) 当前工作流与痛点
- 现有流程（谁、在什么系统、怎么做）
- 人工成本（导表、拼表、汇报、追任务）
- 稳定性与质量问题（格式不稳定、口径变化、结论不可信）
- 组织问题（多角色协同断点、升级机制缺失）

## 3) 场景需求
- 自然语言取数（跨系统、多维筛选、固定口径）
- 主动预警与订阅（阈值、频率、触达人）
- 深度诊断（根因定位、横向/纵向下钻）
- 执行闭环（任务分发、催办、升级、复盘）
- 低风险代操作（可控自动化与人工确认）
- 行业对标（商圈/竞品/品类）

## 4) 数据与技术约束
- 数据源覆盖（POS、外卖、团购、供应链、财务、舆情等）
- 接入方式（API、导入、现有平台）
- 数据准确性要求（100%取数、分析仅作辅助）
- 安全与合规要求（权限、审计、托管）
- 更新与时效要求（日报/周报/准实时）

## 5) 角色差异
- 总部管理层
- 区域/督导
- 门店/执行角色
- 运维/客服

每个角色至少提炼：
- Top 3 高频问题
- Top 3 决策动作
- 关键展示/交互偏好

## 6) 产品优先级
- P0: 没有就无法演示价值的能力
- P1: 明确增益、可在下一版本落地
- P2: 探索项或依赖外部条件

## 7) Demo落地要求（AaaS CoWork）
- 对话先给“结论+动作”，再给数据证据
- 一次对话至少沉淀一种对象：任务/提醒/报告
- 用真实访谈问题或等价改写，避免“空洞问答”
- Mock数据要标注 `Mock`，并可追溯到场景来源
